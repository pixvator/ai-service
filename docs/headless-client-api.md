# Headless Client API

This document describes the client-facing API exposed by the headless AI billing gateway.

The gateway has no dashboard login, admin API, or native configuration API. Channels, provider keys, SMTP settings, OAuth settings, Turnstile settings, users, and schema migrations are managed by the central `new-api` instance that shares the same database.

## Base URL

```text
https://gateway.example.com
```

All JSON endpoints use:

```http
Content-Type: application/json
```

Most management responses use this envelope:

```json
{
  "success": true,
  "message": "",
  "data": {}
}
```

Relay endpoints keep OpenAI-compatible or provider-compatible response formats.

## Authentication

Relay and token lifecycle APIs use an operational API key:

```http
Authorization: Bearer sk-...
```

For token lifecycle APIs, this bearer key also determines which user is being operated on. The gateway validates the key in the `tokens` table, reads its `user_id`, and all `/api/token` operations are scoped to that owner. Clients do not pass a separate `user_id` for token management.

The gateway also accepts provider-compatible key locations on relay routes:

```http
x-api-key: sk-...
x-goog-api-key: sk-...
```

Gemini-compatible routes may also pass:

```text
?key=sk-...
```

Token lifecycle endpoints require an existing valid `sk-...` token for the same user. Bootstrap tokens should be issued by the central service, created by the shared management system, or generated by registration when the shared `GENERATE_DEFAULT_TOKEN` option is enabled.

## Public Status

### `GET /api/status`

Returns the gateway/system status and public runtime capabilities loaded from shared options.

```bash
curl https://gateway.example.com/api/status
```

## Registration

Registration is pure JSON and does not create a browser session.

### `POST /api/verification/email`

Sends an email verification code. If Turnstile is enabled in shared options, `turnstile_token` is required.

Request:

```json
{
  "email": "user@example.com",
  "turnstile_token": "client_turnstile_token"
}
```

Response:

```json
{
  "success": true,
  "message": ""
}
```

The verification code is stored against the encrypted email identifier used by the gateway. The raw email is only used for SMTP delivery.

### `POST /api/user/register`

Creates a user. If email verification is enabled, the `code` must match the email code sent by `/api/verification/email`.

Request:

```json
{
  "username": "alice",
  "password": "strong-password",
  "email": "alice@example.com",
  "code": "123456",
  "turnstile_token": "client_turnstile_token",
  "aff_code": ""
}
```

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "id": 1,
    "username": "alice",
    "display_name": "alice",
    "email": "alice@example.com",
    "role": 1,
    "status": 1,
    "group": "default",
    "default_token": {
      "id": 10,
      "name": "alice的初始令牌",
      "key": "sk-...",
      "expired_time": -1,
      "remain_quota": 500000,
      "unlimited_quota": true,
      "group": "auto"
    }
  }
}
```

Notes:

- Password login is not exposed by this gateway.
- Registration only returns success/failure. It does not set cookies or return a session.
- `default_token` is present only when the shared `GENERATE_DEFAULT_TOKEN` option is enabled.
- If `default_token` is absent, a usable `sk-...` token must be provisioned separately or by shared central logic.
- Email is stored and compared as an encrypted deterministic value, so client-visible `email` fields returned by the gateway are not raw email addresses.

### `GET /api/user/email-exists`

Checks whether a user email already exists. This endpoint does not require Turnstile and does not depend on registration being enabled.

Request:

```http
GET /api/user/email-exists?email=alice@example.com
```

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "exists": true,
    "user": {
      "id": 1,
      "username": "alice",
      "display_name": "alice",
      "email": "encrypted_email_value",
      "role": 1,
      "status": 1,
      "group": "default"
    },
    "default_token": {
      "id": 10,
      "name": "alice的初始令牌",
      "key": "sk-...",
      "expired_time": -1,
      "remain_quota": 500000,
      "unlimited_quota": true,
      "group": "auto"
    }
  }
}
```

When `exists` is `false`, `user` and `default_token` are omitted. `default_token` is also omitted when the user does not have the generated initial token.

## OAuth Registration

OAuth is stateless and returns JSON. It does not create a persistent login session.

### `GET /api/oauth/:provider/login`

Supported built-in provider slugs depend on shared configuration. Common slugs are `github`, `discord`, `oidc`, and `linuxdo`. Custom OAuth provider slugs are loaded from the shared `custom_oauth_providers` table.

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "state": "random_state",
    "redirect_url": "https://provider.example.com/oauth/authorize?..."
  }
}
```

Client flow:

1. Call `/api/oauth/github/login`.
2. Open `data.redirect_url` in the client/browser.
3. Provider redirects to the configured callback URL.
4. Client sends the callback `code` and optional `state` to the callback endpoint below, or lets the callback URL hit the gateway directly.

### `GET /api/oauth/:provider/callback`

Consumes the OAuth provider callback `code`, verifies identity, creates or finds the user, and returns profile data.

Query parameters:

```text
code=provider_code
state=random_state
aff=optional_affiliate_code
```

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "id": 1,
    "username": "alice",
    "display_name": "Alice",
    "email": "alice@example.com",
    "role": 1,
    "status": 1,
    "group": "default",
    "default_token": {
      "id": 10,
      "name": "alice的初始令牌",
      "key": "sk-...",
      "expired_time": -1,
      "remain_quota": 500000,
      "unlimited_quota": true,
      "group": "auto"
    }
  }
}
```

`default_token` is returned only when the OAuth callback creates a new user and the shared `GENERATE_DEFAULT_TOKEN` option is enabled. Existing OAuth users do not receive `default_token`, because the full secret for an existing token cannot be safely recovered later.

## Token Lifecycle

All token lifecycle endpoints require:

```http
Authorization: Bearer sk-...
```

The authenticated bearer token decides the user scope:

```text
Authorization key -> tokens.key -> tokens.user_id -> current user
```

For example, if `sk-user-a-token` belongs to `user_id = 10`, then `GET /api/token` lists only tokens for user 10, `POST /api/token` creates a new token for user 10, and `DELETE /api/token/:id` can only revoke a token owned by user 10.

There is no `user_id` request field on these APIs. Supplying another user's token id in the path will fail because the handler looks up records by both token id and authenticated user id.

### `GET /api/token`

Lists tokens for the authenticated user.

Query parameters:

```text
p=1&page_size=20
```

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "page": 1,
    "page_size": 20,
    "total": 1,
    "items": [
      {
        "id": 10,
        "user_id": 1,
        "key": "abcd**********wxyz",
        "status": 1,
        "name": "desktop-app",
        "expired_time": -1,
        "remain_quota": 500000,
        "unlimited_quota": false,
        "model_limits_enabled": false,
        "model_limits": "",
        "group": "default",
        "cross_group_retry": false
      }
    ]
  }
}
```

### `POST /api/token`

Creates a token for the authenticated user.

Request:

```json
{
  "name": "desktop-app",
  "expired_time": -1,
  "remain_quota": 500000,
  "unlimited_quota": false,
  "model_limits_enabled": false,
  "model_limits": "",
  "allow_ips": "",
  "group": "default",
  "cross_group_retry": false
}
```

Response:

```json
{
  "success": true,
  "message": ""
}
```

After creating a token, call `POST /api/token/:id/key` to fetch the full key once you know the created token id from `GET /api/token`.

### `GET /api/token/:id`

Returns a single masked token owned by the authenticated user.

### `POST /api/token/:id/key`

Returns the full token key.

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "key": "sk-..."
  }
}
```

### `PUT /api/token`

Updates a token owned by the authenticated user.

Request:

```json
{
  "id": 10,
  "name": "desktop-app",
  "expired_time": -1,
  "remain_quota": 500000,
  "unlimited_quota": false,
  "model_limits_enabled": false,
  "model_limits": "",
  "allow_ips": "",
  "group": "default",
  "cross_group_retry": false,
  "status": 1
}
```

To update only token status:

```http
PUT /api/token?status_only=true
```

### `DELETE /api/token/:id`

Deletes/revokes a token owned by the authenticated user.

Response:

```json
{
  "success": true,
  "message": ""
}
```

### `GET /api/usage/token`

Returns usage for the provided token. This endpoint accepts expired or exhausted tokens but still requires the owning user to be enabled.

Response:

```json
{
  "code": true,
  "message": "ok",
  "data": {
    "object": "token_usage",
    "name": "desktop-app",
    "total_granted": 500000,
    "total_used": 1000,
    "total_available": 499000,
    "unlimited_quota": false,
    "model_limits": {},
    "model_limits_enabled": false,
    "expires_at": 0
  }
}
```

## Relay APIs

Relay requests use the same `sk-...` token for authentication and billing. Quota is deducted synchronously when the request completes.

```http

POST /v1/embeddings
```

```bash
curl https://gateway.example.com/v1/chat/completions \
  -H "Authorization: Bearer sk-..." \
  -H "Content-Type: application/json" \
```

## Error Shapes

Management API failures usually return HTTP 200 with:

```json
{
  "success": false,
  "message": "error message"
}
```

Authentication failures may return HTTP 401/403.

Relay failures use OpenAI-compatible error responses where possible:

```json
{
  "error": {
    "message": "invalid token",
    "type": "new_api_error"
  }
}
```

## Client Checklist

- Store `sk-...` securely. Treat it as the billing credential.
- Send relay tokens from the client. Do not replace them with a service key.
- Use `/api/usage/token` to show quota/usage for the current key.
- Use `/api/verification/email` before `/api/user/register` when email verification is enabled.
- Use the central management service for admin/config/channel changes.
